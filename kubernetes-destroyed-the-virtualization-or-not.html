Kubernetes destroyed the virtualization... or NOT

Well ... no

Ok maybe are missing some context

 

I started this topic looking around the data center world compared with the cloud providers.

In the last 5y we saw a cloud rush , a lot of company are sharing the advantage of a cloud migration and how they save money or scale faster ... but is always true what we see?
Most of the times we are looking only builded success case.

Meanwhile in the same period Kubernetes entered in the challenge of most of the online companies.

But why we are talking about cloud / datacenter and kubernetes ?

Today we will take in consideration the main topics of 

  • Virtual machines
  • Data center
  • Kubernetes
  • Cloud approach

I know that the Data center topic should be covered more in deep, maybe talking about the entire ecosystem (db, vertical service, corporate usage , LEGACY!!!!) however today i'd like to explore only the kubernetes and virtualization rather than lose the discussion in thousand of concepts

Long story short

A long time ago Borg was born in Google .. now formally named kubernetes,
in the first period the main evaluation was:
"why the virtualization overhead if we will use kubernetes".

This concept was justified in some way for overlapping of duties,
if i migrate a vm as a kubernetes pod this means i can remove the vm so 
kubernetes > virtual machine.

We can imagine a picture like this looking back of some years

The introduction was good al almost all declared this could be a new way.

Moving faster, all companies changed a bit this picture with an expansion of kubernetes 

Like an infection (good this time) kubernetes come as a main platform for moste of the online company...

In a datacenter , you know, there no something eternal , hardware will go in EOL , the existing is different sometimes between generation and specs , not all nodes probably where reused with se same hw specs and options fiber channel vs 1gbit vs 10gbit)

Is always true , based on the hw configuration that sometimes this encouraged a forced/incorrect usage of kubernetes node labeling in order to bend the infrastructure to better fit the products migrated

Houston we have a problem...

Now the environment is fragmented and even if the technology is the same there are some logical segmentations that create some problems in a day by day:

  • better than label if you need to isolate some workload you should approach a multi-cluster with the same controlplane
  • updates require to manage in a different way looking for the label/role

Moreover an hw fault could create a problem .... and i know in a infrastructure as a code , puppet foreman chef salt ansible terraform , this will be recovered fast ... but is not always true ... you know... some machines/roles in a mid/ent company are still fragile

Other problems are related with new hardware...

  • is the provisioning working as expected ?
  • and if we change the hw vendor ?
  • is the monitoring able to find the new hw as in the previous 
  • etc etc when you have to handle new preseed/ks because the layout is changed ☠️

 

Last but not least ... ok kubernetes but what about the bare metal density ?

Virtualization

  • starting from the lower level of the infrastructure, with a vmware installation we can onboard multiple different certified vendors, this help also de budget creating a saving plan with discounts 
  • having this abstraction layer is quite simple tune the provisioning based on this , rather than bare metal, it's just a metter of roles not to the entire provisioning
  • hardware fault could be easily covered by live migration
  • maintenance and update can be manage with kubespry having a snapshot for fast recover
  • density could be increased with mode nodes in the same hw
  • flexibility to scale up and to upset a new datacenter in case of migration
  • team at the and is able to scale , less is more
